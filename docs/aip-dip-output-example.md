# AIP and DIP output for your transfer (example)

This document walks through the code that builds the AIP and DIP, and shows the **exact output layout** for a transfer like your **mptest_01** (JPG + TIF folders).

---

## Your sample input

```
mptest_01/
├── JPG/
│   ├── Grants_025_1478-1639_ID1579_0001.jpg
│   └── ... (other .jpg files)
└── TIF/
    ├── some_scan_001.tif
    └── ... (other .tif files)
```

You submit it with:

```bash
a3m --name=mptest_01 file:///data/transfers/mptest_01/
```

---

## 1. Where the code runs (high level)

| Step | Where it lives | What it does |
|------|----------------|--------------|
| **Download** | `a3m/client/clientScripts/a3m_download_transfer.py` | Copies your folder into `transfer/<uuid>/objects/` (so paths match the rest of the workflow). |
| **Assign UUIDs** | `assign_file_uuids.py` | Walks `transfer/.../objects/` and creates a `File` (and events) for each file. |
| **Characterize** | `characterize_file.py` | Runs FPR characterization (e.g. ffprobe, exiftool, mediainfo) on each file; stores results in DB. |
| **Normalize** | `normalize.py` | Builds preservation + access derivatives; access copies and thumbnails go under **SIP directory** in `DIP/objects/`. |
| **Create SIP** | `create_sip_from_transfer_objects.py` | Moves transfer `objects/` into the ingest SIP directory and updates DB paths. |
| **METS** | `create_mets_v2.py` | Writes the METS manifest for the SIP (and AIP). |
| **Prepare AIP** | `bag_with_empty_directories.py` | Turns the SIP directory into a **BagIt** bag (payload: `logs/`, `objects/`, `README.html`, `metadata/`, `METS.<uuid>.xml`). |
| **Compress AIP** | `compress_aip.py` | With default config (**uncompressed**), does not create `.7z`; AIP stays a directory. |
| **Verify AIP** | `verify_aip.py` | Validates the bag and checksums, then **moves the AIP** to `share/completed/`. |
| **DIP** | workflow (e.g. `move_or_merge`) | Moves `SIP_directory/DIP` to `share/dips/<SIPDirectoryBasename>/`. |

So: **AIP** = the bag (directory or `.7z`) under `share/completed/`. **DIP** = the DIP tree under `share/dips/`.

---

## 2. Default processing config (what applies to you)

From `a3m/server/processing.py`:

- **normalize** = True → preservation + access derivatives (and thumbnails) are created.
- **generate_dip** = True → DIP is built (access copies, thumbnails, METS copy).
- **aip_compression_algorithm** = UNCOMPRESSED → AIP is a **directory**, not a `.7z` file (unless you change this).

So for your run you get: one **AIP** (directory) and one **DIP** (directory).

---

## 3. Output locations (paths)

a3m uses a single **shared directory** (e.g. `~/.local/share/a3m/share` or, in Docker, `/home/a3m/.local/share/a3m/share`). Under it:

- **AIP** after success:  
  `share/completed/<name>-<sip_uuid>/`  
  Example: `share/completed/mptest_01-37d5b50d-6d70-45bd-afdb-ff7a6847d995/`

- **DIP** after success:  
  `share/dips/<name>-<sip_uuid>/`  
  Example: `share/dips/mptest_01-37d5b50d-6d70-45bd-afdb-ff7a6847d995/`

`<sip_uuid>` is the SIP UUID (same as the AIP identifier). So for one transfer you get one AIP directory and one DIP directory, both named with your transfer name and that UUID.

---

## 4. AIP layout (your example)

The AIP is a **BagIt** bag. With default **uncompressed** setting it looks like this:

```
share/completed/mptest_01-<sip_uuid>/
├── bagit.txt
├── manifest-sha256.txt
├── tagmanifest-sha256.txt
└── data/
    ├── logs/
    │   └── ... (processing logs)
    ├── objects/
    │   ├── JPG/
    │   │   ├── Grants_025_1478-1639_ID1579_0001.jpg
    │   │   └── ...
    │   ├── TIF/
    │   │   ├── some_scan_001.tif
    │   │   └── ...
    │   └── (if normalization ran: preservation/access derivatives under objects/)
    ├── metadata/
    │   └── ... (metadata, transfer metadata, etc.)
    ├── README.html
    └── METS.<sip_uuid>.xml
```

- **data/objects/** = your original **JPG/** and **TIF/** content plus any normalized preservation/access files the workflow puts under `objects/`.
- **data/metadata/** = metadata and transfer metadata.
- **data/logs/** = processing logs.
- **data/METS.<sip_uuid>.xml** = METS manifest for the AIP.
- Bag manifests (`manifest-sha256.txt`, `tagmanifest-sha256.txt`) and `bagit.txt` are generated by `bag_with_empty_directories` and used by `verify_aip`.

If you switch to a compressed algorithm (e.g. 7z), `compress_aip` will turn that directory into a single file, e.g.  
`share/completed/mptest_01-<sip_uuid>.7z`, and that file is what gets moved to `completed/`.

---

## 5. DIP layout (your example)

The DIP is built under the SIP directory and then moved to `share/dips/<name>-<sip_uuid>/`:

```
share/dips/mptest_01-<sip_uuid>/
├── objects/
│   ├── <file_uuid>-<original_name>.<ext>   (access derivatives, e.g. JPEGs from TIFs)
│   └── ... (thumbnails and other access copies)
└── METS.<sip_uuid>.xml
```

- **objects/** = access derivatives (e.g. JPEGs for viewing) and thumbnails. Names are typically like `<file_uuid>-<original_basename>.<ext>`.
- **METS.<sip_uuid>.xml** = copy of the METS manifest for use with the DIP.

So for your **mptest_01** with **JPG** and **TIF** files:

- **AIP** = long-term package: originals (JPG + TIF) plus preservation/technical metadata and logs in a BagIt bag under `share/completed/mptest_01-<sip_uuid>/`.
- **DIP** = dissemination package: access-friendly versions (and thumbnails) under `share/dips/mptest_01-<sip_uuid>/objects/`, plus METS.

## 6. Summary table (your sample)

| Item | Content | Location (example) |
|------|--------|---------------------|
| **Input** | `mptest_01/` with `JPG/*.jpg` and `TIF/*.tif` | Your `file://` path |
| **After download** | Same tree under `objects/` | `.../currentlyProcessing/transfer/<uuid>/objects/{JPG,TIF}/` |
| **AIP** | BagIt bag: originals + metadata + logs + METS | `share/completed/mptest_01-<sip_uuid>/` (dir or `.7z` if compressed) |
| **DIP** | Access derivatives + thumbnails + METS | `share/dips/mptest_01-<sip_uuid>/` |

So: one transfer name (e.g. **mptest_01**) → one **AIP** and one **DIP**, each in the paths above, with the same `<name>-<sip_uuid>` in the directory name.
